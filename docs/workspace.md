# `stuff.workspace` — local isolated workspaces with async exec

> **Status: implemented** in `stuff/workspace.py` (stdlib-only; tests in `tests/test_workspace.py`).
> Exposes `WorkspaceManager`, `Workspace`, `Policy`, `Exec`, `ExecResult`, `Output`, `attach`.
>
> Design notes / thinking doc. This is its **own project**; the only link to
> the security-agent proposal's **G2/G3/G4** (workspace-manager / snapshot / executor) is the
> *concept* — a much smaller, *local, non-cloud* take: no ECS/Fargate, no S3, no STS. One
> stdlib-only Python module that runs **locally on an x86 Linux or macOS machine** and lets a caller
> spin up a contained working directory, run shell command lines in it asynchronously, and tear it
> down cleanly.
>
> **Cross-platform reality up front:** the strong Linux isolation primitives — bubblewrap,
> network namespaces, cgroups — are **Linux-only**. macOS has no equivalent that works the same way;
> it gets `sandbox-exec` (Seatbelt) at best and the portable rlimits-and-process-group floor
> otherwise. The API is identical on both; the *strength* differs and is reported honestly (§4).

## 1. What it's for

A **workspace** is *an isolated working directory + an environment + resource & policy limits*, in
which you can execute shell command lines as child processes — `python3 …`, a custom executable, a
pipeline of tools — without those processes stepping on each other, the host, or leaving a mess.

Concretely, the three driving use cases:

1. **Event-triggered scripts.** A `ubon_cstuff` event (a detection, a zone breach, a scene change)
   fires a shell script. The script runs in a fresh, bounded workspace so a runaway or a crash is
   contained and the artifacts it produces land in a known place.
2. **Pulling data from `ubon_cstuff` storage.** A script fetches clips/crops/embeddings from the
   storage API into the workspace, processes them, and emits a result.
3. **Arbitrary compute.** Run `python3`, run a bespoke binary, chain tools with a pipe — the
   workspace is just where it happens, with caps on it.

### Non-goals (v1)

- **Not a hard multi-tenant security boundary.** See the threat model (§3) — this contains
  *accidents* and bounds *blast radius* for first-party code. It is not a VM and does not claim to
  stop determined hostile code. The upgrade path to that is real and noted, but out of scope now.
- **No cloud anything.** No remote scheduler, no object-store snapshots, no credential broker.
- **No agent loop, no LLM, no capability registry.** That's the security-agent's G1/G5/G6 world;
  this module is just the *executor + lifecycle* primitive those would sit on top of.

## 2. Threat model (state it first — it drives every other decision)

The scripts that run here are **first-party**: written by the operator, or generated by an agent
*under the operator's control and review*. They are not arbitrary code submitted by an untrusted
third party.

Given that, what we actually want:

| Goal | Why | Mechanism |
|---|---|---|
| **Containment of accidents** | a buggy script shouldn't `rm -rf` the host or fill the disk | dedicated dir as CWD; filesystem bind-scoping; file-size & disk caps |
| **Bounded blast radius** | a runaway shouldn't take the box down | CPU / memory / wall-clock / pids rlimits; kill the whole process tree |
| **Reproducibility & cleanup** | every run starts clean and leaves nothing behind | create → run → destroy; idempotent teardown |
| **Optional egress cut** | an event script usually has no business phoning home | `--unshare-net` (bwrap) or netns |
| **Auditability** | "what ran, with what limits, what came out" | per-exec transcript record |

What we explicitly do **not** promise in v1: defence against a malicious process trying to *escape*.
If/when external-authored code runs here, the kernel tier (§4) escalates to gVisor / nsjail / a
microVM. Designing the API so that swap is invisible to callers is a requirement (§4); shipping it
is not.

**This is the one decision I most want you to confirm** — that first-party-trusted is the right
threat model for now. Everything below follows from it.

## 3. Shape of the API

Module: `stuff.workspace`. Sketch (names/types are the proposal, not final):

```python
from stuff.workspace import WorkspaceManager, Policy

mgr = WorkspaceManager(root="/var/lib/ubon/workspaces")   # base dir owning all workspaces

ws = mgr.create(
    label="evt-1234",                       # human tag; id is a uuid regardless
    policy=Policy(
        sandbox       = "auto",             # "auto" | "bwrap" | "none"
        network       = False,              # cut egress when sandboxed
        allow_cmds    = None,               # None = any on PATH; or {"python3","ffprobe",...}
        env           = {"UBON_DB": "/data/storage"},  # explicit; otherwise a scrubbed minimum
        cpu_seconds   = 60,                 # rlimit RLIMIT_CPU
        mem_bytes     = 2 << 30,            # rlimit RLIMIT_AS
        wall_seconds  = 120,                # parent-side timeout -> kill tree
        pids          = 256,                # rlimit RLIMIT_NPROC
        file_bytes    = 1 << 30,            # rlimit RLIMIT_FSIZE
        max_output    = 64 * 1024,          # per-stream cap before spilling to a file
        ro_binds      = ["/usr", "/bin", "/lib", "/lib64", "/data/storage"],
        rw_binds      = [],                 # extra writable mounts beyond the ws dir
    ),
    ttl=600,                                # optional lease; sweeper destroys on expiry
)

# --- async by construction: exec() returns a handle immediately, never blocks ---
h = ws.exec("python3 analyze.py --since 10m")   # str (via shell) or argv list
...                                              # caller keeps working
if h.poll() is not None:                         # finished? non-blocking
    ...
res = h.wait(timeout=30)                          # block (optionally bounded) -> ExecResult
print(res.returncode, res.stdout_text(), res.duration, res.timed_out)

mgr.destroy(ws.id)                                # full teardown (idempotent)
```

Core objects:

- **`WorkspaceManager`** — owns a `root` dir, a registry of live workspaces, and a background
  sweeper (TTL expiry + orphan GC). `create()`, `get(id)`, `destroy(id)`, `list()`, and the
  snapshot hooks (§7). One manager per process is the norm; it's safe to construct several over
  disjoint roots.
- **`Workspace`** — `id`, `dir` (Path), `policy`, `expires`. Methods: `exec(...)`, `path(*parts)`
  (resolve a path inside the ws), `write(name, data)` / `read(name)` convenience, `execs()` (live
  handles). Supports `with mgr.create(...) as ws:` → auto-destroy on exit.
- **`Exec`** (handle) — `pid`, `poll() -> int|None`, `wait(timeout=None) -> ExecResult`,
  `running`, `returncode`, `kill()`, plus streaming access (`stdout`/`stderr` as growing files).
- **`ExecResult`** — `returncode`, `stdout`/`stderr` (`Output`: inline bytes ≤ cap, else a file
  ref — *never silently truncated*), `duration`, `timed_out`, `killed`, `seq` (transcript index).

I deliberately mirror `subprocess`/`Popen` ergonomics (`poll`/`wait`/`returncode`) so it feels
familiar and is trivial to reason about.

## 4. Isolation model — tiered, degrades honestly

The same API runs at three strengths; `sandbox="auto"` picks the strongest available and records
which it used (so the audit line never lies about what protected a run).

| Tier | OS | Mechanism | Gets you | Needs |
|---|---|---|---|---|
| `none` | Linux + macOS | `subprocess` in the ws dir, new session (`start_new_session=True`), `setrlimit` via `preexec_fn`, scrubbed env | cleanup, resource caps, tree-kill | nothing — fully portable |
| `bwrap` | **Linux only** | wrap argv in **bubblewrap**: bind ws dir rw, system paths ro, private `/tmp`, `--unshare-net` (if `network=False`), `--die-with-parent`, `--new-session` | the above **+ filesystem scoping + namespace isolation + egress cut**, all *unprivileged* | `bwrap` on PATH (Linux user-ns) |
| `seatbelt` | **macOS only** | wrap argv in **`sandbox-exec`** with a generated profile (deny file-write outside the ws, optionally deny network) | filesystem scoping + egress cut, no root | `sandbox-exec` (ships with macOS; *deprecated* but functional) |
| *(future)* `nsjail` / `gvisor` / microVM | Linux | stronger kernel boundary | defence against escape (untrusted code) | heavier deps; deferred per §2 |

`auto` resolves to `bwrap` on Linux, `seatbelt` on macOS, falling back to `none` only if asked
(see "degrades loudly" below). The honest summary: **strong unprivileged sandboxing on Linux;
best-effort on macOS** (Seatbelt is deprecated and its profile language is finicky, so macOS may in
practice run `none` + rlimits — fine for the first-party threat model, but stated plainly).

Design rules:

- **The tier is a policy field, not a code path callers see.** `ws.exec(...)` is identical whether
  it's wrapped by bwrap, sandbox-exec, or nothing. Moving Linux → gVisor later must not touch caller
  code, and the *same script* runs unmodified on a Mac laptop and a Linux box.
- **Resource caps are always on** (rlimits work at every tier, on both OSes). The sandbox adds the
  *spatial* isolation; rlimits add the *quantitative* isolation. They're complementary.
- **`auto` degrades loudly, not silently.** If an explicit tier is requested but unavailable, that's
  an error at `create()` time. `auto` picks the best for the OS and **stamps the chosen tier** on
  the workspace and every `ExecResult`, so a Mac run that fell back to `none` is visible in the logs
  rather than masquerading as sandboxed.
- **cgroups v2 (Linux only)** is the right way to get a *hard* memory/CPU/pids cap (rlimits are
  per-process, so a fork bomb evades `RLIMIT_AS`). Where a cgroup is delegated to us, the manager
  can place each workspace in its own sub-cgroup. A strengthening of the Linux tiers, not a new
  tier; absent on macOS (rlimits only there).

## 5. Async / concurrency model

The requirement is "asynchronous, with being able to check/wait for completion." Two realities to
serve: `ubon_cstuff` event callbacks are likely **synchronous** C-binding calls, while some callers
live in an **asyncio** loop. I propose one core + one façade:

- **Core: thread-backed, sync-friendly.** `exec()` launches a `Popen` and starts (daemon) reader
  threads that drain stdout/stderr into bounded ring buffers, spilling to a file past `max_output`.
  The handle is usable from any thread: `poll()` is non-blocking, `wait(timeout)` blocks the caller
  only, the process runs regardless. This works in a plain sync event callback with no event loop.
- **Façade: asyncio.** `await ws.run(cmd)` and an `AsyncExec` with `await h.wait()` built on
  `asyncio.create_subprocess_exec`, for callers already in a loop. Same `ExecResult`.

Either way: **many execs per workspace are allowed** (they share the dir); **each exec is its own
process group/session** so `kill()` and timeout reliably take down the whole tree (the recurring
bug class in this codebase has been *not* cleaning the whole tree — so tree-kill is a first-class,
tested guarantee, not an afterthought).

Timeouts: `wall_seconds` (policy) and a per-exec `timeout=` both arm a parent-side timer that
escalates `SIGTERM` → grace → `SIGKILL` to the group, and marks `ExecResult.timed_out`.

## 6. Output handling

Bounded by default, **never silently truncated** (lifting the one genuinely good idea from the G4
contract). Each stream: up to `max_output` bytes inline in the result; beyond that, the full stream
spills to `ws/.io/<exec-seq>.{out,err}` and `Output.ref` points at it with `spilled=True`. Callers
that expect big output read the file; callers that expect a line read `.stdout_text()`. This avoids
both OOM-by-logging and the silent-data-loss footgun.

## 7. Lifecycle, persistence, and the "teardown" guarantee

- **`create`** — mkdir `root/<id>/`, lay out `work/` (CWD), `.io/` (spill), `state.json`
  (id, policy, tier, created, expires, pid-table), register in the manager.
- **`destroy(id, reason="")`** — **idempotent, total teardown**: tree-kill every live exec, then
  `rmtree` the dir, then drop from the registry. Calling it twice, or on an unknown id, is a no-op,
  not an error. This is the *teardown+recreate* principle the rest of this codebase has learned the
  hard way — there is no in-place "reset"; you destroy and create fresh.
- **TTL + sweeper** — a background thread destroys workspaces past `expires` (optionally
  snapshotting first), and on manager startup **GCs orphans**: dirs under `root` whose owning
  process is gone (detected via `state.json` + a pidfile / flock). A crash never leaks disk.
- **Snapshot (G3-lite, optional)** — `mgr.snapshot(id) -> path` writes `tar.gz(dir) + manifest`
  (file list, sizes, sha256, originating policy minus secrets, reason). `mgr.restore(path) -> ws`
  materialises it into a *new* workspace. Useful for the resume/escalation idea, but I'd gate it
  behind a real need — for pure event-triggered scripts it's likely YAGNI in v1. Flag for you.

## 8. Integrating with `ubon_cstuff` (kept *out* of the module)

The module ships in the general `../stuff` library, so it must **not depend on `ubon_cstuff`**.
Integration is a usage pattern the caller wires up, three ways, cleanest first:

1. **Data in by file.** The trigger fetches what the script needs *before* exec and writes it into
   the ws (`ws.write("clip.mp4", bytes)` / a crop / an embeddings `.npy`), then runs the script with
   the path. The script stays storage-agnostic and trivially testable.
2. **Data in by env/args.** Pass `UBON_DB=/data/storage`, a signed time-boxed ref, a track id, a
   time range as env/argv; the script pulls what it needs via a small first-party fetch tool that
   *is* on the `allow_cmds` list (e.g. a `ubon-fetch` CLI). This is the analogue of the proposal's
   "media by reference, not by copy."
3. **Event → exec glue (example).**

```python
mgr = WorkspaceManager(root="/var/lib/ubon/workspaces")

def on_event(evt):                      # called from a ubon_cstuff event callback (sync)
    ws = mgr.create(label=f"evt-{evt.id}",
                    policy=Policy(network=False, wall_seconds=60,
                                  allow_cmds={"python3", "ubon-fetch"},
                                  env={"UBON_DB": "/data/storage"}))
    h = ws.exec(["python3", "/opt/ubon/scripts/on_breach.py",
                 "--track", str(evt.track_id), "--t", f"{evt.t:.3f}"])
    # fire-and-forget: a sweeper/whendone hook destroys ws when h completes (or TTL hits)
    h.when_done(lambda res: (log_result(evt, res), mgr.destroy(ws.id)))
```

`when_done(cb)` (a thin callback on the reader-thread completion) keeps the trigger non-blocking and
ties cleanup to completion — no caller bookkeeping. (For asyncio callers, `await ws.run(...)` then
`mgr.destroy`.)

## 9. Failure modes I want to get right by construction

- **Process-tree leaks** → every exec in its own session; `kill()`/timeout signal the group;
  destroy kills all; bwrap `--die-with-parent` as backstop. Tested explicitly (fork a child that
  forks, assert nothing survives destroy).
- **Disk leak on crash** → pidfile + startup orphan GC; `file_bytes` rlimit; optional disk quota.
- **Output OOM** → bounded buffers + spill (§6).
- **Zombie / double-wait** → handle owns the `Popen`; `wait()` is idempotent and thread-safe.
- **`auto` lying about protection** → tier recorded on ws + every result; degrade is explicit.

## 9a. Interactive mode (PTY) — implemented

Interactive terminals / "ssh-like" sessions are supported via a **pseudo-terminal**, built in layers:

- **`ws.exec(cmd, tty=True)`** allocates a PTY; the child's stdio is the slave and (via `setsid` +
  `TIOCSCTTY`) it's the controlling terminal, so REPLs, `bash -i`, `vim`, progress bars, etc. behave
  exactly as at a real terminal. The handle exposes `send(bytes)`, `read()`, `resize(rows, cols)`,
  and `master_fd`.
- **`ws.shell()`** — convenience for an interactive shell in the workspace.
- **`attach(handle)`** — bridges *your* controlling terminal (raw mode) to the PTY: a real terminal
  window onto the workspace command, Ctrl-] to detach. Useful from a CLI.
- **Web terminal** — `master_fd` is a plain fd; pipe it to a websocket and point xterm.js at it. This
  is an integration concern (lives in the app, e.g. `edge_node`), not in the general module.
- **SSH** — deliberately *not* an sshd inside the sandbox (defeats `--unshare-net`, adds key/auth
  surface). The ssh-like experience is a long-lived PTY session you attach/detach to; for
  reconnect-survival, run `tmux`/`screen` as the workspace command and attach to it. Genuine remote
  access = the host's own sshd + a `workspace attach` command, not sshd-per-workspace.

Verified working: a live `python3 -i` REPL driven over the PTY (echoed prompts, incremental output),
and `bash -i`.

## 10. Decisions — resolved

1. **Threat model:** first-party-trusted (contain accidents + bound blast radius + cleanup +
   optional egress cut + audit), **not** escape-proof. Ship bwrap/seatbelt + rlimits now; gVisor/
   nsjail remain a future drop-in kernel tier behind the same API.
2. **macOS:** `sandbox-exec` (Seatbelt) is the Mac tier — generated allow-default profile denying
   file-write outside the ws and (optionally) network. Best-effort and honestly labelled; falls back
   to `none`+rlimits if unavailable. **Strong isolation is guaranteed only on Linux.**
3. **Linux strong tier:** `bwrap`, but selected via a **capability probe** (`_bwrap_usable()`), not
   mere presence — bwrap installed-but-unable (userns blocked, e.g. inside another sandbox) means
   `auto` → `none` and explicit `bwrap` raises. (Found the hard way: this build machine blocks
   nested userns, so bwrap can't run *here* though it's installed — exactly the case the probe
   handles.)
4. **cgroups v2 (Linux):** implemented for **hard** mem/pids caps, but **opt-in**
   (`WorkspaceManager(cgroups=True)`) and off by default — enabling it *relocates the manager process
   into a child cgroup* (cgroup v2's no-internal-processes rule), a process-global side effect to
   consent to. Needs a delegated cgroup (`systemd-run --user --scope -p Delegate=yes`, or root);
   degrades to rlimits otherwise, recorded not silent. (A re-nest guard was added after an early
   version stacked `.manager/.manager/…` across managers.)
5. **Async surface:** the **sync, thread-backed handle** (`exec()` → `poll`/`wait`/`when_done`/
   `kill`) is the primary — it satisfies "asynchronous + check/wait" from a synchronous
   `ubon_cstuff` event callback with no event loop. An asyncio façade can wrap it later if a loop
   caller wants `await`; not needed for v1.
6. **Snapshot/restore:** included (lightweight `tar.gz` + manifest of file hashes) since it's cheap
   and enables the resume/escalation idea; not load-bearing for the event-script path.

## 10b. Original open questions (kept for record)

1. **Threat model (§2):** confirm first-party-trusted is right for now (drives whether we ship just
   the sandbox+rlimits tiers, or invest in nsjail/gVisor up front).
2. **macOS isolation expectation (§4):** is best-effort on Mac acceptable — i.e. `sandbox-exec` if
   it behaves, else `none`+rlimits — with strong isolation only guaranteed on Linux? Or does macOS
   need a real filesystem/network boundary on day one (which pushes toward a Linux VM/container even
   on the Mac, contradicting "stdlib-only, local")?
3. **bwrap requirement on Linux** — fine to require `bubblewrap` for the strong Linux tier (it's a
   one-line `apt`/`brew`-of-Linux install), with `none` as the no-install floor?
4. **cgroups v2 (§4):** worth wiring up the Linux hard-cap path now, or are per-process rlimits
   enough for v1 given first-party scripts?
5. **Async surface (§5):** is the sync thread-backed handle enough for the `ubon_cstuff` event path,
   or do you want the asyncio façade in v1 too?
6. **Snapshot/restore (§7):** in scope for v1, or defer until a resume/escalation use case is real?
7. **Doc/module home:** I put this at `stuff/docs/workspace.md` (co-located with where the module
   will live). Say if you'd rather it sat elsewhere.

## 11. Rough module layout (when we build it)

Single file `stuff/workspace.py` (stdlib only — `subprocess`, `os`, `signal`, `resource`,
`threading`, `shutil`, `pathlib`, `tarfile`, `hashlib`, `uuid`, `json`, `platform`; `asyncio` only
for the façade), exported from `stuff/__init__.py` as `WorkspaceManager`, `Workspace`, `Policy`,
`Exec`, `ExecResult`. Tests cover: clean create/destroy, tree-kill, rlimit trip, output spill,
timeout, sandbox-present vs absent (per-OS), orphan GC. `resource`/rlimits, process groups, and
`subprocess` all work on both Linux and macOS, so the portable `none` tier is genuinely portable;
the sandbox wrapper is selected by `platform.system()`. One dependency-free file matches the rest of
`stuff` and runs on a Mac laptop or an x86 Linux box with nothing to install but (optionally)
`bubblewrap` on Linux.
